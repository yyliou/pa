Making R Fast

Measure Before You Optimize

The cardinal rule: never guess where time goes — measure it.

system.time({
  x <- numeric(0)
  for (i in 1:20000) x <- c(x, i)   # the growing-vector sin
})
   user  system elapsed 
  0.277   0.010   0.287 
system.time({
  y <- numeric(20000)
  for (i in 1:20000) y[i] <- i      # pre-allocated
})
   user  system elapsed 
  0.001   0.000   0.002

Same loop, wildly different cost — because c() re-copies the whole vector every iteration.

Profiling

For whole programs, the profiler attributes time to functions:

slow_demo <- function(n) {
  m <- matrix(rnorm(n * 50), ncol = 50)
  out <- numeric(ncol(m))
  for (j in seq_len(ncol(m))) out[j] <- mean(sort(m[, j]))
  out
}
Rprof(tmp <- tempfile())
invisible(slow_demo(20000))
Rprof(NULL)
head(summaryRprof(tmp)$by.self, 5)
              self.time self.pct total.time total.pct
".doSortWrap"      0.02       50       0.02        50
"rnorm"            0.02       50       0.02        50
  • Rprof() samples the call stack; summaryRprof() tabulates it.
  • Rprof(memory.profiling = TRUE) and tracemem() expose hidden copying.

Vectorize

The single biggest R speedup: replace element-by-element loops with whole-vector operations, which run in compiled C:

v <- rnorm(1e6)
system.time({ s <- 0; for (xi in v) s <- s + xi^2 })
   user  system elapsed 
  0.022   0.000   0.021 
system.time(sum(v^2))
   user  system elapsed 
  0.002   0.000   0.002
  • rowSums(), colMeans(), cumsum(), pmax(), ifelse()-free logical math ((x > 0) * x) all vectorize common patterns.
  • The apply family is clearer than loops but not inherently faster — true vector ops are.

Lookup Performance

How you find things matters as data grows:

n <- 1e5
named.vec <- setNames(rnorm(n), paste0("k", 1:n))
e <- new.env(hash = TRUE)
for (k in c("k50000")) assign(k, named.vec[[k]], envir = e)
system.time(for (i in 1:200) named.vec["k50000"])   # name lookup: linear
   user  system elapsed 
  0.088   0.001   0.089 
system.time(for (i in 1:200) get("k50000", envir = e))  # hash: constant
   user  system elapsed 
  0.000   0.000   0.001
  • Name-based vector indexing scans; hashed environments (and the fastmatch/data.table packages) jump straight there.
  • match() beats repeated %in% in loops; sorted keys enable findInterval().

Memory Matters

R keeps everything in RAM, and copying is the silent killer:

gc()[, c("used", "max used")] |> head(2)
          used max used
Ncells  736405  1420447
Vcells 2586977  8387712
object.size(rnorm(1e6))
8000048 bytes
  • rm() big objects you are done with; gc() reports (and triggers) garbage collection.
  • Modify columns rather than whole data frames inside loops; each “small” change of a big object may copy all of it.
  • For data bigger than RAM: data.table (in-memory but frugal), ff/bigmemory (disk-backed), databases (Chapter 11), or Hadoop (Chapter 26).

The Byte-Code Compiler

R compiles functions to byte code; ensure hot functions benefit:

library(compiler)
f <- function(n) { s <- 0; for (i in 1:n) s <- s + i; s }
fc <- cmpfun(f)
system.time(f(2e6)); system.time(fc(2e6))
   user  system elapsed 
  0.035   0.001   0.034 
   user  system elapsed 
  0.033   0.000   0.033

Since R 3.4 the JIT compiles most functions automatically — but the gap shows what the interpreter costs, and why C++ via the Rcpp package remains the heavy artillery for unavoidable loops.

Parallel Computation

Many tasks are embarrassingly parallel — independent pieces, no communication:

library(parallel)
detectCores()
[1] 8
system.time(res1 <- lapply(1:8, function(i) mean(rnorm(4e5))))
   user  system elapsed 
  0.076   0.001   0.077 
cl <- makeCluster(2)
system.time(res2 <- parLapply(cl, 1:8, function(i) mean(rnorm(4e5))))
   user  system elapsed 
  0.001   0.000   0.048 
stopCluster(cl)
  • mclapply() (Unix/macOS forking) is the one-line version; parLapply() works everywhere.
  • Simulations, bootstraps, cross-validation folds, and grid searches parallelize perfectly; overhead means small tasks may not.

An Optimization Checklist

  1. Measure (system.time, Rprof) — find the real bottleneck.
  2. Vectorize the hot loop; pre-allocate results.
  3. Avoid copies — beware growing objects and repeated subsetting of big frames.
  4. Better algorithms/structures — hashing, sorting, data.table.
  5. Parallelize independent work.
  6. Drop to C++ (Rcpp) only when all of the above still isn’t enough.

Readable-but-fast beats clever-but-fragile: optimize the 5% that matters and leave the rest clear.
