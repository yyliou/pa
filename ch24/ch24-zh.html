讓 R 更快

先量測、再優化

黃金法則：絕不盲目優化。先找出時間與記憶體實際花在哪，再修那裡。R 兩者都有工具。

量測效能

計時：system.time

最簡單的量測——包住任何運算式：

system.time(for (i in 1:1e6) sqrt(i))
   user  system elapsed 
  0.015   0.000   0.015

三個數字：user time（R 本身做的工）、system time（OS 替 R 做的工）、elapsed time（牆上時鐘）。gcFirst=TRUE（預設）先做垃圾回收，讀數更乾淨。（嚴謹的微基準請用現代的 microbenchmark 與 bench，會重複並比較。）

剖析：Rprof 與 summaryRprof

要看整支程式時間花在哪，就剖析（profile）它：

Rprof("Rprof.out", interval = 0.02, memory.profiling = FALSE)
do.stuff()                       # ……你的程式……
Rprof(NULL)                      # 停止剖析
summaryRprof("Rprof.out")        # 哪些函式吃掉時間

Rprof 每 interval 秒採樣一次呼叫堆疊、寫進檔案；Rprof(NULL) 停止。summaryRprof 依自身與總時間排名（加 memory= 還能依配置量）。這就是你找出那一行佔掉大半執行時間的方法——往往出人意料。

監測記憶體

gc()                             # 垃圾回收「並」回報用量
          used (Mb) gc trigger (Mb) limit (Mb) max used (Mb)
Ncells  626460 33.5    1420424 75.9         NA  1420424 75.9
Vcells 1158800  8.9    8388608 64.0      16384  2013854 15.4
object.size("Hello world!")      # 單一物件的約略大小
120 bytes
  • gc() 強制垃圾回收並回報目前記憶體用量。
  • object.size(x) 估計物件佔用——當一個「小」資料框其實龐大時，價值連城。
  • memory.profile() 依物件型別拆解用量；Windows 上 memory.size()／memory.limit() 回報並設定總用量上限。
  • Rprofmem(filename, threshold=) 記錄大配置（R 須以記憶體剖析編譯）。

優化你的程式碼

善用向量運算

R 最大的提速槓桿：把逐元素的迭代程式碼換成整個向量的運算。第八章的例子加強版——在迴圈中擴張向量 vs. 一次處理：

slow <- function(n) { v <- NA; for (i in 1:n) v[i] <- i^2; v }
fast <- function(n) { (1:n)^2 }
system.time(slow(50000))["elapsed"]
elapsed 
  0.007 
system.time(fast(50000))["elapsed"]
elapsed 
      0

向量版又快又清楚。兩個原因：迴圈主體要重複分派 R 函式上萬次，而擴張向量每一步都重新配置記憶體。預先配置（v <- numeric(n)）有幫助；向量化幫助更大。

把問題改寫成內建函式

許多迴圈能改寫成以 C 實作的函式呼叫：

  • apply 家族——sapply、lapply、vapply、mapply、tapply（第九、十二章）；
  • 矩陣代數——%*%、crossprod、rowSums、colMeans 取代巢狀迴圈；
  • outer、cumsum、diff、tabulate、findInterval 處理常見模式。
m <- matrix(1:1e6, ncol=1000)
system.time(apply(m, 2, sum))["elapsed"]      # 易讀
elapsed 
  0.005 
system.time(colSums(m))["elapsed"]            # 內建，快得多
elapsed 
  0.001

教訓：在 C 裡跑迴圈的內建函式，勝過任何 R 層級的迴圈。

查值效能：用環境當雜湊表

反覆以名稱查值？具名向量或串列做的是線性掃描；環境是雜湊表，查找近乎常數時間。要做上千次按鍵查找，就把資料放進環境：

e <- new.env()
assign("BG", 42, envir=e)        # 或 e[["BG"]] <- 42
get("BG", envir=e)               # O(1) 查找
[1] 42

書中把它與具名向量查找對比，發現表越大、環境越是大幅勝出——許多高效能 R 套件背後的訣竅。

位元組碼編譯器，以及更遠

R 內附位元組碼編譯器（compiler 套件），能加速純 R 函式——而且自 R 2.14 起，base 與 recommended 套件已預先編譯；自 R 3.4 起，函式在使用時即時（JIT）編譯：

library(compiler)
slow.c <- cmpfun(slow)           # 編譯成位元組碼
system.time(slow.c(50000))["elapsed"]
elapsed 
  0.005

Tip

優化的階梯。 (1) 向量化。(2) 用內建／C 函式與矩陣代數。(3) 預先配置；查找用環境。(4) 編譯熱點函式。(5) 之後才平行化（第二十六章）或用 C++（Rcpp）重寫瓶頸。每一步後都重新剖析——並在「夠快」時停手。
