讓 R 跑得快

先量測，再優化

鐵則：永遠不要猜時間花在哪——量它。

system.time({
  x <- numeric(0)
  for (i in 1:20000) x <- c(x, i)   # 「向量長大」之罪
})
   user  system elapsed 
  0.276   0.011   0.288 
system.time({
  y <- numeric(20000)
  for (i in 1:20000) y[i] <- i      # 預先配置
})
   user  system elapsed 
  0.001   0.000   0.001

同一個迴圈、成本天差地遠——因為 c() 每一輪都把整個向量重新複製一次。

效能剖析

對整支程式，剖析器把時間歸給各函式：

slow_demo <- function(n) {
  m <- matrix(rnorm(n * 50), ncol = 50)
  out <- numeric(ncol(m))
  for (j in seq_len(ncol(m))) out[j] <- mean(sort(m[, j]))
  out
}
Rprof(tmp <- tempfile())
invisible(slow_demo(20000))
Rprof(NULL)
head(summaryRprof(tmp)$by.self, 5)
        self.time self.pct total.time total.pct
"rnorm"      0.02       50       0.02        50
"sort"       0.02       50       0.02        50
  • Rprof() 抽樣呼叫堆疊；summaryRprof() 把結果列表。
  • Rprof(memory.profiling = TRUE) 與 tracemem() 揭露看不見的複製。

向量化

R 最大的單一加速來源：把逐元素迴圈換成整向量運算，後者跑在編譯過的 C 裡：

v <- rnorm(1e6)
system.time({ s <- 0; for (xi in v) s <- s + xi^2 })
   user  system elapsed 
  0.022   0.000   0.021 
system.time(sum(v^2))
   user  system elapsed 
  0.002   0.000   0.002
  • rowSums()、colMeans()、cumsum()、pmax()，以及不靠 ifelse() 的邏輯運算（(x > 0) * x）都把常見模式向量化了。
  • apply 家族比迴圈清楚，但本質上不更快——真正的向量運算才快。

查找效能

資料一大，「怎麼找東西」就開始要緊：

n <- 1e5
named.vec <- setNames(rnorm(n), paste0("k", 1:n))
e <- new.env(hash = TRUE)
for (k in c("k50000")) assign(k, named.vec[[k]], envir = e)
system.time(for (i in 1:200) named.vec["k50000"])   # 名稱查找：線性掃描
   user  system elapsed 
  0.088   0.000   0.089 
system.time(for (i in 1:200) get("k50000", envir = e))  # 雜湊：常數時間
   user  system elapsed 
  0.000   0.000   0.001
  • 依名稱索引向量是掃描；雜湊環境（以及 fastmatch/data.table 套件）直接跳到目標。
  • 迴圈裡 match() 勝過反覆 %in%；排序過的鍵可用 findInterval()。

記憶體很重要

R 把一切放在 RAM，而複製是沉默的殺手：

gc()[, c("used", "max used")] |> head(2)
          used max used
Ncells  736405  1420447
Vcells 2586890  8387625
object.size(rnorm(1e6))
8000048 bytes
  • 用完的大物件 rm() 掉；gc() 回報（並觸發）垃圾回收。
  • 迴圈內修改欄位、別修改整個資料框；對大物件的每個「小」改動都可能整份複製。
  • 資料大過 RAM 時：data.table（仍在記憶體但省）、ff/bigmemory（磁碟支援）、資料庫（第十一章）、Hadoop（第二十六章）。

位元組碼編譯器

R 會把函式編成位元組碼；確保熱點函式受惠：

library(compiler)
f <- function(n) { s <- 0; for (i in 1:n) s <- s + i; s }
fc <- cmpfun(f)
system.time(f(2e6)); system.time(fc(2e6))
   user  system elapsed 
  0.034   0.000   0.034 
   user  system elapsed 
  0.033   0.000   0.033

R 3.4 起 JIT 會自動編譯多數函式——但這個差距顯示了直譯器的成本，也說明為何躲不掉的迴圈最終要靠 Rcpp 套件的 C++ 重砲。

平行運算

許多工作是「尷尬地平行」——彼此獨立、無需溝通：

library(parallel)
detectCores()
[1] 8
system.time(res1 <- lapply(1:8, function(i) mean(rnorm(4e5))))
   user  system elapsed 
  0.075   0.003   0.078 
cl <- makeCluster(2)
system.time(res2 <- parLapply(cl, 1:8, function(i) mean(rnorm(4e5))))
   user  system elapsed 
  0.001   0.001   0.049 
stopCluster(cl)
  • mclapply()（Unix/macOS 的 fork）是一行版；parLapply() 各平台通用。
  • 模擬、拔靴、交叉驗證折、網格搜尋都完美平行；但開銷意味著太小的工作未必划算。

優化檢查清單

  1. 量測（system.time、Rprof）——找出真正的瓶頸。
  2. 向量化熱點迴圈；預先配置結果。
  3. 避免複製——當心長大的物件與對大資料框的反覆切片。
  4. 更好的演算法／結構——雜湊、排序、data.table。
  5. 平行化獨立的工作。
  6. 以上都不夠時才下探 C++（Rcpp）。

可讀又快勝過聰明而脆弱：優化那要緊的 5%，其餘保持清楚。
