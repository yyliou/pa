用 R 處理大數據

為什麼用 Hadoop

有些問題大到單機處理不來。解法：把工作拆成片段、平行求解、再合併——洗衣比喻：十台洗衣機洗十堆，跟一台洗一堆一樣快。許多統計方法天生可平行——bagging、boosting、隨機森林（第二十、二十一章）各自獨立配每棵樹。

書中年代，Hadoop 是大數據的事實標準：儲存龐大資料集、跨叢集運算、容忍機器故障。本章示範如何從 R 駕馭 Hadoop。

Warning

這是 2012 年的快照。 Hadoop 的主導地位已淡去——今日的大數據工作偏向 Spark（R 介面 sparklyr）、雲端資料倉儲，以及在單機處理超出記憶體資料的 arrow／duckdb。但下面的概念——Map/Reduce、分散式儲存——仍是基石，其中數項至今仍支撐著現代系統。

Hadoop 怎麼運作

Map/Reduce

Hadoop 的程式模型把每個工作拆成兩步：

  • Map——獨立處理每筆紀錄，產出鍵–值對。（剖析每行網頁日誌；產出 (地點, 1)。）
  • Reduce——蒐集每個鍵的所有值並合併。（按地點加總計數。）

因為每個 map 獨立執行，工作能散到你擁有的所有機器上。書中的經典範例：網頁流量報表、按地點的流量、預測使用者行為——全都能表達成先 map 後 reduce。

兩大支柱

Map/Reduce 之外，Hadoop 還提供：

  • 分散式資料儲存——HDFS（Hadoop Distributed File System）：把檔案切成區塊、跨機器複製以容錯，並呈現為單一邏輯檔案系統。
  • 叢集管理——跨伺服器排程工作、重啟失敗的任務、把運算搬到資料所在處。

框架本身以 Java 寫成，但你不必寫 Java——R 能驅動它。

從 R 駕馭 Hadoop

三條路

書中介紹三種從 R 使用 Hadoop 的方式：

  • RHadoop——最成熟、整合最好的 R 套件組；
  • Segue——在 Amazon Elastic MapReduce 上跑 R 的 lapply 式工作；
  • Hadoop streaming——透過標準輸入輸出，把任意 R 腳本當 mapper／reducer 餵給 Hadoop。

RHadoop

RHadoop 是三個套件：

  • rmr——核心：用 R 寫 Map/Reduce 工作、進出資料；
  • rhdfs——從 R 管理 HDFS 上的檔案（多數一般檔案操作）；
  • rhbase——介接 Hadoop 的資料庫 HBase。

先設定、再下工作（rmr 語法，以未執行形式呈現——它需要 Hadoop 叢集）：

library(rmr2)
# 按性別計數死亡：一個 map 步、一個 reduce 步，皆為 R 函式
result <- mapreduce(
  input = "deaths.csv",
  map    = function(k, v) keyval(v$sex, 1),
  reduce = function(sex, counts) keyval(sex, sum(counts)))
from.dfs(result)                 # 把結果拉回 R

書中的四節點範例統計 2009 年美國各性別死亡數——一個跨叢集平行執行的工作。to.dfs／from.dfs 在 R 與 HDFS 間搬資料。

雲端上的 Hadoop

不必自己擁有叢集——用租的。書中走過 Amazon Elastic MapReduce（EMR）：啟動叢集、用範本設定、跑工作、拆掉——只付用掉的分鐘數。Segue 把這套包起來，讓 R 使用者幾乎不必懂 Hadoop 就能在 EMR 上跑平行 lapply。

# Segue 風格：在租來的叢集上做平行 lapply（概念示意）
library(segue)
cluster <- createCluster(numInstances=4)
results <- emrlapply(cluster, my.list, my.function)
stopCluster(cluster)

把它放進脈絡

Tip

歷久彌新的教訓。 Hadoop 這個產品在退場，但它的想法無所不在：(1) 切分資料與運算；(2) 把工作表達成 map（獨立、平行）後 reduce（合併）；(3) 把運算搬向資料，而非反過來；(4) 在雲端租用彈性算力。現代 R 透過 sparklyr、arrow、duckdb，以及平行化的 future／furrr 觸及大數據——但你會在它們之中都認出 Map/Reduce。
