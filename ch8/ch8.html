Where Names Live

Symbols and Bindings

Defining a variable really means binding a symbol to a value inside an environment:

x <- 1            # bind symbol `x` to 1 in the current environment
y <- quote(x + 1) # an unevaluated expression mentioning `x`
eval(y)           # evaluate it: `x` is looked up NOW
[1] 2
x <- 100
eval(y)           # same expression, new answer
[1] 101

An expression is data until evaluated; the value of its symbols is decided at evaluation time, in some environment.

Delayed Evaluation and Promises

Evaluation can be postponed deliberately:

delayedAssign("lazy", {message("computing now!"); 42})
# nothing printed yet ...
lazy             # first touch forces the promise
[1] 42

Note

Promises. A promise packages an expression plus the environment to evaluate it in, and computes its value only when first needed. R passes all function arguments as promises — that is “lazy evaluation,” and it is why an unused argument can contain an error without crashing the call.

What Is an Environment?

An environment is a set of symbol-to-value bindings plus a pointer to a parent environment. Lookup walks the chain: current environment first, then parent, then grandparent, until emptyenv().

e <- new.env()
assign("z", 99, envir = e)
get("z", envir = e)
[1] 99
exists("z", envir = e)
[1] TRUE
ls(envir = e)
[1] "z"

Other tools: rm(..., envir=) removes bindings; environmentName() labels well-known environments.

The Global Environment

Each session starts in the global environment — your workspace:

environment()             # where am I?
<environment: R_GlobalEnv>
ls()[1:5]                 # what have I defined?
[1] "e"    "lazy" "x"    "y"    NA

The chain above the global environment consists of the attached packages — exactly the search path:

search()
[1] ".GlobalEnv"        "package:stats"     "package:graphics" 
[4] "package:grDevices" "package:utils"     "package:datasets" 
[7] "package:methods"   "Autoloads"         "package:base"

This is why loading a package makes its functions visible: it inserts an environment into the lookup chain.

Environments and Functions

Calling a function creates a fresh environment for its local variables, whose parent is the environment where the function was defined (not where it was called):

make_counter <- function() {
  count <- 0
  function() {
    count <<- count + 1   # write to the enclosing environment
    count
  }
}
tick <- make_counter()
tick(); tick(); tick()
[1] 1
[1] 2
[1] 3

The inner function remembers count between calls — a closure. This lexical scoping rule is the heart of R’s function semantics.

Global vs. Lexical Assignment

v <- "global"
f <- function() { v <- "local"; v }
g <- function() { v <<- "overwritten"; v }
f(); v          # local assignment leaves global alone
[1] "local"
[1] "global"
g(); v          # <<- climbed up and replaced it
[1] "overwritten"
[1] "overwritten"

Warning

Use <<- sparingly. Functions that silently modify state outside themselves are hard to test and debug. Closures (previous slide) are the disciplined use case.

Working with the Call Stack

The environment where a function was called is reachable too — essential for functions that interpret expressions in their caller’s context:

whodunit <- function() sys.function(-1)   # who called me?
show_caller_x <- function() {
  eval(quote(x), envir = parent.frame())
}
x <- "caller's x"
show_caller_x()
[1] "caller's x"
  • sys.call() — the current call; match.call() — with argument names filled in.
  • parent.frame() — the calling environment. Functions like lm() use exactly this trick to evaluate formulas against your data.

Environments in Packages

  • Each loaded package contributes an environment to the search path; lookup explains shadowing — a name in your workspace hides one in a package.
  • Disambiguate explicitly with the :: operator:
stats::sd(c(1, 2, 3, 4))
[1] 1.290994
  • Packages use namespaces so their internal helpers stay invisible, and promises (via lazy data loading) so big data sets cost nothing until touched.
